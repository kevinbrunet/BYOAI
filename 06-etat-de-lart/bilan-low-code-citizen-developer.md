# État de l'art — Bilan low-code / citizen developer (recherche web, juillet 2026)

## Ce que disent les chiffres

- ~ Gartner prévoyait que 70 % des nouvelles applications seraient construites sur des plateformes low-code/no-code d'ici fin 2025 (contre <25 % en 2020) — prédiction vendue partout, à re-sourcer vers le communiqué Gartner original avant citation.
- ~ 31 % des entreprises utilisant le no-code/low-code n'ont construit **aucune** de leurs applications à plus forte valeur avec — le chiffre clé du désenchantement : l'outil sert pour le périphérique, pas pour ce qui compte.
- ~ Plus de 75 % des responsables IT citent la sécurité comme préoccupation n°1 du citizen development.
- ✓ Le discours du secteur a muté : plus personne ne promet le remplacement des développeurs ; le récit 2025-2026 est "collaboration citizen/professional developers" et "l'IA va enfin faire tenir la promesse du low-code".

## Lecture critique pour la série

1. **Le bilan conforte l'objection ET la réponse** de [destin-low-code-no-code](../04-objections/destin-low-code-no-code.md) : l'adoption est réelle mais cantonnée au non-critique (31 %) — cohérent avec le plafond de contraintes : dès que le besoin est spécifique/critique, on sort de l'espace exprimable.
2. **Attention au retournement d'argument** : le secteur low-code dit désormais "l'IA + low-code = citizen development enfin viable". Le livre doit expliquer pourquoi IA + code pur ≠ IA + low-code : l'IA dans le low-code génère *dans* l'espace contraint (elle hérite du plafond) ; l'IA en code pur n'a pas de plafond. Sans ce paragraphe, un lecteur assimilera le BYOIA à la n-ième promesse citizen developer.
3. **Vocabulaire** : éviter "citizen developer" dans le livre — le terme est chargé d'échec. Le [développeur-harnais](../01-concepts/equipe-agile-developpeur-harnais.md) est structurellement différent (un professionnel rapproché du terrain, pas un amateur outillé) : c'est un argument, pas juste un choix de mot.
